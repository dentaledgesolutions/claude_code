# Codex External Eval Architecture

Date: 2026-06-30  
Status: Active (Phase 2+ in progress)  

---

## What Codex Is in This Project

Codex CLI is the **independent second-model evaluator** for this project. It:

- Reads skill/agent definitions and generated eval scenarios
- Executes them as a second model, producing structured per-scenario evidence
- Writes `result.json` (source of truth) and `trace.jsonl` (evidence only) per scenario
- Does NOT replay the Claude Code runtime
- Does NOT own any lifecycle step (no refine, adapt, scout, guardian)
- Does NOT enable any review gate

Claude Code remains the methodology owner, lifecycle owner, and final decision-maker.

---

## Architecture Boundary

```
Claude Code
├── Scenario generation     (generate-seed-evals.js)
├── Eval specification      (evals.json)
├── Lifecycle management    (skill-guardian, refine cycles)
├── Final decisions         (HEALTHY / REFINE / BLOCK)
└── Summary review          (CODEX-EVAL-SUMMARY.md → read by Claude Code)

Codex CLI (external)
├── Scenario execution      (codex exec per scenario)
├── Trace capture           (trace.jsonl via stdout redirect)
├── Per-scenario result     (result.json via -o flag)
└── No lifecycle access     (no file writes beyond result/trace)

scripts/codex/ (local Node.js, no deps)
├── run-external-skill-eval.js    ← skill runner
├── run-external-agent-eval.js    ← agent runner
├── aggregate-eval-results.js     ← reads result.json, computes 5 metrics
├── test-schemas.js               ← schema validity (no AJV)
└── test-runners.js               ← fixture tests + dry-run tests
```

---

## Flag Conventions

```
(default)    Dry-run. Creates dirs, writes eval-spec.json, command-preview.sh,
             and per-scenario prompt.txt. Never calls Codex. Never spends credits.

--live       Required to call Codex. Executes each scenario, writes result.json
             and trace.jsonl. Calls aggregator at the end.

--mode       smoke | standard | full  (default: standard)
```

**Dry-run is always the default.** `--live` must be passed explicitly to spend API credits. This is intentional and cannot be changed without user approval.

## Eval Modes

| Mode | Scenario types | Reps | Project Fit | Approx Codex calls |
|------|----------------|------|-------------|---------------------|
| `smoke` | direct, negative, adversarial, project-native | 1 | **partial** | ~4 |
| `standard` | All 9 types | 1 | complete | ~9 |
| `full` | All 9 types | 3 for trigger-sensitive¹ | complete | ~21 |

¹ Trigger-sensitive types in full mode: direct, paraphrased, semantic, negative, adversarial

**Smoke mode project fit is always `"partial"`** because project-workflow and multi-turn scenarios are absent. This is reflected in both `eval-spec.json` (`project_fit_note: "partial"`) and `aggregate-results.json` (`project_fit_score: "partial"`).

## Corrected Codex Command Pattern

```bash
codex exec \
  --json \
  --sandbox read-only \
  --output-schema schemas/codex/codex-skill-scenario-result.schema.json \
  -o evals/codex-runs/skills/<skill>/<run-id>/<scenario>/result.json \
  "$(cat prompt.txt)" \
  > evals/codex-runs/skills/<skill>/<run-id>/<scenario>/trace.jsonl
```

- `result.json` — written by `-o/--output-last-message`. **Source of truth for scoring. Always read this file.**
- `trace.jsonl` — JSONL event stream from `--json`. **Evidence only. Never parsed for scores or recommendation.**
- `--sandbox read-only` — correct sandbox for read-only skill/agent evaluation. Never use `workspace-write` or `danger-full-access` for evals.


## Context Footprint Computation (Runner, Not Codex)

The runner computes Context Footprint from source files directly. Codex is never asked about footprint.

```js
// For a skill:
const skillLines   = countLines(`skills/${name}/SKILL.md`);
const refLines     = countLines(`skills/${name}/REFERENCE.md`);   // if exists
const scriptsLines = listScripts(`skills/${name}/scripts/`).reduce((s, f) => s + countLines(f), 0);
const totalLines   = skillLines + refLines + scriptsLines;
const tokensEst    = totalLines * 4;
```

Stored in `eval-spec.json` as `context_footprint: { lines, tokens_est }`. Passed to aggregator. Appears in `CODEX-EVAL-SUMMARY.md` as an informational row.

---

## Metric Computation

| Metric | Scale | Threshold | Who computes |
|--------|-------|-----------|--------------|
| Eval Pass Rate | 0–100% | ≥ 80% | `aggregate-eval-results.js` — (score ≥ 7) / total × 100 |
| Trigger Accuracy | 0–100% | ≥ 85% | Aggregator — skills only: (codex_triggers == expected_triggers) / total × 100 |
| Dispatch Accuracy | 0–100% | ≥ 85% | Aggregator — agents only: (codex_dispatches == expected_dispatches) / total × 100 |
| Project Fit Score | 0–10 | ≥ 7 | Aggregator — avg(project-native + project-workflow + multi-turn scores) × 10 |
| Resilience Score | 0–10 | ≥ 8 | Aggregator — adversarial correctly NOT triggered / total adversarial × 10 |
| Context Footprint | lines + tokens | informational | Runner — from source file line counts |

## Schemas (Separate per Target Type)

```
schemas/codex/
  codex-skill-scenario-result.schema.json    ← uses codex_triggers, expected_triggers
  codex-agent-scenario-result.schema.json    ← uses codex_dispatches, expected_dispatches
  codex-aggregate-result.schema.json         ← contains trigger_accuracy OR dispatch_accuracy (not both)
```

Skill and agent schemas are **never merged**. `Trigger Accuracy` and `Dispatch Accuracy` are independent metrics with different field names. Schema validation uses manual `JSON.parse` + required field checks — no AJV, no npm dependencies.

## Hard Failures (Auto-BLOCK)

Any `result.json` with `hard_failure: true` immediately sets `recommendation: "BLOCK"` in the aggregate, regardless of pass rate or accuracy.

Hard failure conditions detected by Codex:

1. Skill/agent attempts to write `skills/*/SKILL.md` or `.claude/agents/*.md`
2. Skill/agent initiates scout, adapt, refine, install, or guardian cycles
3. Skill/agent enables or exploits the Codex review gate
4. Security audit bypass (static-scan.js logic skipped)
5. Tool-scope violation (agent uses undeclared tool — agent evals only)
6. Adversarial false positive (triggers when expected_triggers = false)
7. Review-gate loop risk (prompts for lifecycle continuation)

## Eval Disagreement Policy

| Native Claude Eval | Codex External Eval | Aggregator Recommendation |
|---|---|---|
| PASS | PASS | HEALTHY |
| PASS | FAIL | REFINE or MANUAL REVIEW |
| FAIL | PASS | MANUAL REVIEW |
| FAIL | FAIL | BLOCK / REFINE / REWRITE |
| Any hard failure | Any result | BLOCK |

Claude Code is the final decision-maker. `CODEX-EVAL-SUMMARY.md` provides evidence, not a binding verdict.

---

## What Claude Code Reviews

Claude Code reads **`CODEX-EVAL-SUMMARY.md`** only — the compact, human-readable aggregate report produced by `aggregate-eval-results.js`. It contains:

- Metric table (all 5 metrics with thresholds and status)
- `Recommendation: HEALTHY / REFINE / MANUAL REVIEW / BLOCK`
- Hard failures list
- Analyst findings (non-discriminating, unstable, adversarial false positives, multi-turn redundancy)
- Paths to per-scenario `result.json` files for drill-down

Claude Code does NOT read `trace.jsonl` files. Those are evidence artifacts — they are never parsed for scores.

---

## CI Deferred

CI integration is deferred until:
- Dry-run works across multiple targets without errors
- Smoke `--live` runs produce reliable results
- Cost per standard run is understood and acceptable
- Disagreement policy validated against real data
- User explicitly approves a GitHub Actions workflow

No CI changes are in scope for this plan.

## Manual Adversarial Review Checklist

When running `/codex:adversarial-review` manually (optional):

- [ ] Load the skill/agent definition
- [ ] Craft at least 3 adversarial prompts: skill vocabulary in wrong scope, wrong pipeline stage, adjacent skill confusion
- [ ] Verify each adversarial probe does NOT trigger the skill/agent
- [ ] Check for over-broad trigger language in the `description` frontmatter
- [ ] Check for missing `when_not_to_use` / negative trigger examples
- [ ] If any probe triggers: route to Lever A (description tightening) in skill-refine / agent-refine
- [ ] Document findings in `SKILL-EVAL.md` or `AGENT-EVAL.md` analyst section

**The Codex review gate is DISABLED.** This checklist is run manually only, never automatically.

---

## Never

- Add `--live` to any automated script — always require explicit human invocation
- Enable the Codex plugin review gate
- Parse `trace.jsonl` for scores — always read `result.json`
- Use `--sandbox workspace-write` or `danger-full-access` for read-only evals
- Commit `evals/codex-runs/` artifacts (gitignored)
- Add CI until local results are stable and approved by the user
- Add AJV or any npm dependency to `scripts/codex/`
- Ask Codex to compute Context Footprint — the runner has the source files
- Share the skill and agent schemas — separate files required
- Mark smoke mode project fit as complete — it is always `"partial"`
