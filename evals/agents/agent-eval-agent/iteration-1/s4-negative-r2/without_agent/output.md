# What agent-eval-agent Does and How It Works

The `agent-eval-agent` is a metrics-driven quality measurement agent for Claude Code agent definitions. Its job is to tell you whether an agent is actually doing its job well — not by reading the agent's instructions and reasoning about whether they seem good, but by running structured test scenarios and scoring real outputs.

## What problem it solves

When you write or adapt a Claude Code agent (a `.md` file in `.claude/agents/`), you have no objective way to know if its `description` field is precise enough to trigger dispatch at the right times, or if it over-dispatches on inputs it should ignore. `agent-eval-agent` solves this by running a controlled experiment: it fires the same prompts both WITH the agent dispatched and WITHOUT (using only general capabilities), then measures the delta.

## The core mechanism: parallel dispatch pairs

For each test scenario, `agent-eval-agent` spawns two subagents simultaneously:

- A **with-agent** subagent that dispatches the agent under test and records its output
- A **baseline** subagent that handles the same prompt using only general Claude capabilities

This pair structure is what makes the scoring meaningful — it isolates the agent's contribution from what Claude can do on its own.

## The 9 scenario types

Scenarios are generated by `generate-agent-evals.js` and cover:

- **dispatch-type** (run 3 reps each for stability): direct triggers, paraphrased triggers, semantic triggers, negative cases (should NOT dispatch), and adversarial cases (attempts to trick the agent into dispatching)
- **workflow-type** (run once): edge cases, project-native scenarios, project-workflow scenarios, and multi-turn conversations

When `--context evals/project-context.json` is provided, 3 additional project-specific scenarios are added (totaling 9), which is always preferred.

## The 5 metrics

| Metric | What it measures | Threshold |
|---|---|---|
| Eval Pass Rate | % of scenarios where composite score ≥ 7 | ≥ 80% |
| Dispatch Accuracy | % of correct dispatch decisions across all dispatch-type reps | ≥ 85% |
| Context Footprint | Agent file size in lines/estimated tokens | informational |
| Project Fit Score | How well the agent uses project terminology and artifact paths | ≥ 7/10 |
| Resilience Score | How reliably the agent resists adversarial over-dispatch | ≥ 8/10 |

## Scoring rules

Each scenario gets a composite score (0–10) built from dispatch correctness (40%), workflow completeness (30%), and output format quality (30%). Adversarial scenarios are binary: score 10 if the agent correctly refused to dispatch, score 0 if it dispatched when it shouldn't have.

Multi-turn scenarios apply a 3-point deduction if the agent re-asks for context that was already provided in the prompt preamble.

## The analyst pass

After scoring, `agent-eval-agent` runs an analyst pass that flags specific problems:

- **UNSTABLE**: dispatched in only 1 or 2 of 3 reps (flaky description wording)
- **ADVERSARIAL_FAILURE**: over-dispatched on an adversarial prompt (description too broad — Lever A)
- **REGRESSION**: the agent performed worse than no agent at all (delta < -2)
- **TOOL_VIOLATION**: agent used a tool not declared in its frontmatter `tools:` field

## What it produces

- `.claude/agents/<name>-EVAL.md` — the full report with metric table, scenario results, analyst observations, and a recommendation (HEALTHY / REFINE / REWRITE / RE-ADAPT)
- `evals/agents/<name>/iteration-N/` — per-scenario output directories with with-agent and without-agent outputs
- `evals/agents/<name>/refine-input.json` — if any metric is below threshold, this handoff file tells `agent-refine-agent` exactly which metrics failed, which scenarios caused the failures, and which lever (A through E) to pull first

## How it fits in the pipeline

`agent-eval-agent` sits between `agent-adapt` and `agent-refine` in the pipeline. It can also be invoked mid-loop by `agent-refine-agent` to score mutations during the autoresearch cycle. When invoked programmatically by the refiner, it adapts to training-set-only mode and signals completion with `EVAL_COMPLETE` on the final line of its output.

In short: it is the objective quality gate that prevents unmeasured agents from being deployed or refined blindly.
